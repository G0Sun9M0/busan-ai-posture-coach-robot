# 바른자세 코치봇 학습일지

> **ESP32-S3 · Xiaozhi AI 4족 자세 코치 로봇**
> 2026년 부산 AI 교육에서 만든 임베디드 AI 로봇 프로젝트의 학습 기록.
> "부산 교육에서 **배운 것**"과 "AI(Claude)와 **협업하며 만들면서 배운 것**"을 함께 정리했다.

- **작성자**: 고성모
- **기간**: 2026.06 (부산 AI 교육)
- **개발 환경**: macOS · ESP-IDF v5.5.4 · C / C++ · FreeRTOS
- **보드**: ESP32-S3
- **AI 프레임워크**: Xiaozhi(小智, MCP 기반 챗봇 펌웨어) 이식

---

## 1. 프로젝트 개요 — 무엇을 만들었나

**바른자세 코치봇**은 센서로 사용자의 자세(앞으로 숙임)를 감지하면, **AI 음성**으로 자세를 교정해 주는 4족 로봇이다.

```
[자세 센서] --숙임 감지--> [ESP32-S3] --판단--> [AI 음성 코칭 + 로봇 반응(다리 모션)]
```

- 입력: 자세 감지(마이크/센서 이벤트)
- 처리: ESP32-S3에서 상태 판단 → Xiaozhi AI로 전송
- 출력: 스피커 음성 안내 + 서보로 4족 다리 반응

핵심은 **"하드웨어를 바닥부터 하나씩 올려서(bring-up) 통합했다"**는 점이다. 처음부터 완성품을 만든 게 아니라, 보드 점등 → 화면 → 서보 → 보행 → 소리 → 마이크 → AI 반응 순서로 **한 단계씩 확실히 동작을 검증한 뒤** 다음으로 넘어갔다.

## 2. 하드웨어 구성 & 통신

| 부품 | 역할 | 통신 / 제어 |
|---|---|---|
| **ESP32-S3** | 메인 MCU | FreeRTOS 태스크 |
| **OLED 디스플레이** | 상태 표시 | I2C |
| **PCA9685** | 16채널 PWM 서보 컨트롤러 | I2C (주소 `0x40`, SDA=GPIO1, SCL=GPIO2, 100kHz) |
| **서보 ×N** | 4족 다리 관절 | PCA9685 PWM |
| **마이크** | 음성/자세 입력 | I2S |
| **스피커** | AI 음성 출력 | I2S |

## 3. 부산 교육에서 배운 핵심 개념 (학습 내용)

### 3.1 ESP-IDF & FreeRTOS
- **ESP-IDF**: Espressif의 공식 IoT 개발 프레임워크. `idf.py build / flash / monitor`로 빌드·플래시·시리얼 모니터를 다룬다.
- **FreeRTOS**: ESP32는 실시간 OS 위에서 동작한다. `app_main()`이 진입점이고, `vTaskDelay()`로 태스크를 재운다. 가장 처음 배운 `00_board_test`가 바로 이 구조(반복 출력)였다.
- **CMake 기반 빌드**: 각 프로젝트는 `CMakeLists.txt` + `main/` 구조. `sdkconfig`로 보드/기능 설정.

### 3.2 I2C — 서보를 움직이는 저수준 통신
- **I2C**는 SDA(데이터)·SCL(클럭) 2선으로 여러 장치를 주소(`0x40` 등)로 구분해 통신하는 방식.
- 라이브러리에 의존하지 않고 `driver/i2c`로 **PCA9685 레지스터를 직접 써서** 서보 각도(PWM 듀티)를 명령하는 법을 배웠다.
- 서보 각도 → PWM 펄스 폭 → PCA9685 레지스터 값으로 변환하는 과정을 이해했다.

### 3.3 I2S — 디지털 오디오
- **I2S**는 마이크/스피커 같은 디지털 오디오 장치와 주고받는 전용 통신 규격.
- `i2s_std` 드라이버로 마이크 입력(캡처)과 스피커 출력(재생)을 구성했다.

### 3.4 4족 보행 (Gait)
- 서보 각도를 **순서대로(시퀀스) 바꿔** 다리를 들고-옮기고-내리는 동작을 반복하면 보행이 된다.
- 단순히 각도를 주는 것과, 그것을 **타이밍에 맞춰 엮어 걷게 하는 것**은 다른 문제라는 걸 배웠다.

### 3.5 AI 음성 (Xiaozhi 프레임워크)
- **Xiaozhi**는 ESP32용 MCP 기반 AI 챗봇 펌웨어. 디바이스 상태 머신, `mcp_server`, OTA(무선 업데이트) 등으로 구성된다.
- 자세 감지 이벤트를 AI에 전달해 음성 반응을 만드는 흐름을 연결했다.

## 4. AI(Claude)와 협업하며 만들면서 배운 것

이 프로젝트는 **Claude Code와 협업(바이브 코딩)**하며 진행했다. 코드를 AI가 빠르게 써 주는 만큼, 내 역할은 **무엇을 왜 만들지 정하고, 결과를 검증·디버깅·통합**하는 데 있었다.

### 4.1 단계적 브링업 전략 — 가장 큰 배움
> "한 번에 다 만들지 말고, 한 단계가 확실히 동작한 뒤 다음으로."

AI에게 전체를 한 번에 시키면 어디서 고장났는지 알 수 없다. 그래서 **각 하드웨어를 독립 프로젝트(00~06)로 쪼개** 하나씩 검증했다. 이 전략 덕분에 "서보는 되는데 보행이 안 된다" 같은 문제를 **범위를 좁혀** 잡을 수 있었다.

### 4.2 크로스플랫폼 이식 (Windows → macOS)
- 원래 교육 자료·Xiaozhi 가이드는 **Windows 기준**이었는데, 내 개발 환경은 macOS였다.
- 경로·툴체인·포트 이름(`/dev/cu.usbmodem*`) 차이를 하나씩 맞춰 **맥에서 ESP-IDF v5.5.4로 빌드 성공**시켰다.
- 이 과정에서 "가이드를 그대로 따라 하는 것"과 "환경 차이를 이해하고 옮기는 것"의 차이를 배웠다.

### 4.3 재현 가능한 실행 환경 만들기
- 매번 손으로 포트를 찾고 플래시하는 게 번거로워, `demo.sh` · `flash_robot.sh` · `flash_sensor.sh`로 **포트 자동 탐색 → 플래시 → 모니터**를 스크립트화했다.
- "한 번 되게 만드는 것"과 "다시 재현 가능하게 만드는 것"은 다르다는 걸 체감했다.

## 5. 단계별 브링업 기록 (00 → 06)

| 단계 | 폴더 | 배운 것 / 검증한 것 |

## 7. 회고 & 앞으로 학습할 것

### 배운 것
- 소프트웨어만이 아니라 **하드웨어를 바닥부터 직접 올려본** 경험. I2C·I2S·PWM 같은 저수준 통신을 손으로 다뤄봤다.
- **문제를 잘게 쪼개 검증하는 습관** — 임베디드에서 특히 강력하다.
- AI 협업으로 빠르게 완주하되, **동작 검증·디버깅은 결국 내 몫**이라는 것.

### 아직 부족한 것 (정직하게)
- I2C/I2S를 "동작하게" 했지만, **버스 타이밍·클럭·레지스터 맵의 원리**까지 완전히 이해한 건 아니다.
- 4족 보행이 걷긴 하지만, **기구학(다리 각도 계산)** 은 조사 수준이고 직접 유도해보진 못했다.
- FreeRTOS **태스크 스케줄링·우선순위**의 깊은 이해가 필요하다.

### 학습 계획
1. **C 포인터·메모리** 기본기 (임베디드의 뿌리)
2. **I2C/I2S 프로토콜 원리** — 데이터시트를 직접 읽고 레지스터 이해
3. **FreeRTOS** 태스크·큐·세마포어 개념
4. **로봇 기구학·PID 제어** — 조사에서 구현으로

> **원칙**: 바이브 코딩으로 만든 코드를 한 줄씩 다시 읽고, 이해 안 되는 개념은 따로 공부해 노트로 남긴 뒤, **도구 없이 축소판을 다시 구현**해 검증한다.

## 8. 실행 방법 (재현)

```bash
# ESP-IDF 환경 활성화
source ~/.espressif/tools/activate_idf_v5.5.4.sh
unset IDF_TARGET

# 데모 실행 (펌웨어가 이미 구워진 경우 — 모니터만)
bash ~/Desktop/0629_busan_ai/demo.sh     # 종료: Ctrl + ]

# 특정 단계 테스트 빌드·플래시 (예: 서보)
cd ~/Desktop/0629_busan_ai/02_servo_test
idf.py -p /dev/cu.usbmodem* flash monitor
```

> 참고: `firmware/`는 Xiaozhi 오픈소스 펌웨어를 이식한 것이며, `managed_components/`와 `build/`는
> 저장소에서 제외했다(용량). `idf.py reconfigure`로 의존성을 다시 받은 뒤 빌드하면 된다.
