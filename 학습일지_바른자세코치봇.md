# 바른자세 코치봇 학습일지

> **ESP32-S3 · Xiaozhi AI 자세 코치 로봇**
> 2026년 부산 AI 교육에서 만든 임베디드 AI 로봇 프로젝트의 학습 기록.
> "부산 교육에서 **배운 것**"과 "AI(Claude)와 **협업하며 만들면서 배운 것**"을 함께 정리했다.

- **작성자**: 고성모
- **기간**: 2026.06 (부산 AI 교육)
- **개발 환경**: macOS · ESP-IDF v5.5.4 · C / C++ · FreeRTOS
- **보드**: ESP32-S3
- **AI 프레임워크**: Xiaozhi(小智, MCP 기반 챗봇 펌웨어) 이식

---

## 1. 프로젝트 개요 — 무엇을 만들었나

**바른자세 코치봇**은 **기울기 센서**로 사용자의 자세(앞으로 숙임)를 감지하면, **AI 음성**으로 자세를 교정해 주는 로봇이다.

```
[기울기 센서] --숙임 감지--> [ESP32-S3] --판단--> [AI 음성 코칭 (마이크/스피커 + Xiaozhi AI)]
```

- 입력: **기울기 센서**로 자세(앞으로 숙임) 감지
- 처리: ESP32-S3에서 상태 판단 → Xiaozhi AI로 전송
- 출력: I2S 스피커로 **AI 음성 안내**

> ⚠️ **중요 — 보행 기능은 없다.** 원래 실습 키트는 **4족 보행 로봇**(서보 기반)이었지만,
> "바른자세 코치"라는 목적에는 **다리로 걷는 기능이 전혀 필요 없었다.**
> 그래서 **서보모터를 제거하고 보행 하드웨어·코드를 데드(dead)로 정리**했다. 이 판단 자체가
> 이 프로젝트에서 배운 중요한 부분이다(→ 4장·7장).

## 2. 하드웨어 구성 & 통신

### 실제로 사용한 것
| 부품 | 역할 | 통신 / 제어 |
|---|---|---|
| **ESP32-S3** | 메인 MCU | FreeRTOS 태스크 |
| **기울기 센서** | 자세(숙임) 감지 | 입력 |
| **INMP441** | I2S 마이크 | I2S (WS=GPIO4, SCK=GPIO5, SD=GPIO6) |
| **MAX98357A** | I2S 앰프(스피커) | I2S (DIN=GPIO7, BCLK=GPIO15, LRC=GPIO16) |
| OLED | 상태 표시(선택) | I2C |

> 마이크·스피커는 I2S 포트 하나(`I2S_NUM_0`)를 **번갈아** 사용한다(동시에 켜지 않음).

### 제거한 것 (데드 하드웨어)
| 부품 | 원래 용도 | 왜 제거했나 |
|---|---|---|
| **PCA9685 + 서보 ×N** | 4족 보행(키트 원본) | 자세 코치엔 보행이 불필요 → 제거 |

## 3. 부산 교육에서 배운 핵심 개념 (학습 내용)

### 3.1 ESP-IDF & FreeRTOS
- **ESP-IDF**: Espressif 공식 IoT 프레임워크. `idf.py build / flash / monitor`로 빌드·플래시·시리얼 모니터.
- **FreeRTOS**: 진입점은 `app_main()`, `vTaskDelay()`로 태스크 제어. 첫 테스트 `00_board_test`가 이 구조였다.
- **CMake 빌드**: 각 프로젝트 = `CMakeLists.txt` + `main/`, `sdkconfig`로 설정.

### 3.2 I2S — 디지털 오디오 (실제 사용)
- **I2S**는 마이크/스피커 같은 디지털 오디오 장치 전용 통신 규격.
- `i2s_std` 드라이버로 **INMP441 마이크 캡처**와 **MAX98357A 스피커 재생**을 구성.
- 마이크 입력 레벨(볼륨)을 계산해 일정 기준(`MIC_TRIGGER_LEVEL`)을 넘으면 반응하도록 실습(`06_mic_reaction_test`).

### 3.3 I2C & 서보 (학습은 했으나 최종 미사용)
- **I2C**(SDA/SCL 2선, 주소로 장치 구분)로 **PCA9685**(16채널 PWM) 서보 컨트롤러를 직접 제어하는 법을 배웠다.
- 서보 각도 → PWM 듀티 → 레지스터 값 변환, 나아가 각도 시퀀스로 **4족 보행 게이트**까지 실습해봤다.
- **다만 최종 코치봇에는 넣지 않았다** — 목적상 불필요해서. (개념 학습으로서의 가치는 남는다.)

### 3.4 기울기 센서 — 자세 감지
- **기울기 센서**로 사용자가 앞으로 숙였는지(자세 무너짐)를 감지하는 것이 이 로봇의 핵심 입력이다.
- "센서를 앞으로 숙이면 자세 경고 + AI 음성"이 동작하도록 연결했다.

### 3.5 AI 음성 (Xiaozhi 프레임워크)
- **Xiaozhi**는 ESP32용 MCP 기반 AI 챗봇 펌웨어(디바이스 상태 머신·`mcp_server`·OTA).
- 자세 감지 이벤트를 AI에 전달해 음성 반응을 만드는 흐름을 연결.

## 4. AI(Claude)와 협업하며 만들면서 배운 것

이 프로젝트는 **Claude Code와 협업(바이브 코딩)**하며 진행했다. 코드를 AI가 빠르게 써 주는 만큼, 내 역할은 **무엇을 왜 만들지 정하고, 결과를 검증·디버깅·통합**하는 데 있었다.

### 4.1 "불필요한 하드웨어를 걷어내는" 설계 판단 — 가장 큰 배움
원래 키트는 4족 보행 로봇이라 서보·PCA9685·보행 코드가 딸려 있었다. 하지만 **"바른자세 코치"에는 걷는 기능이 필요 없었다.**
- 처음엔 서보·보행 테스트(`02`,`03`)까지 해봤지만, 목적에 맞춰 **보행 하드웨어(서보)를 과감히 제거**했다.
- 남길 것(자세 감지·음성)과 버릴 것(보행)을 구분하는 것 — 이게 **기능을 더하는 것보다 중요한 결정**이라는 걸 배웠다.

### 4.2 단계적 브링업 전략
AI에게 전체를 한 번에 시키면 어디서 고장났는지 모른다. 그래서 **각 하드웨어를 독립 프로젝트(00~06)로 쪼개** 하나씩 검증했다.

### 4.3 크로스플랫폼 이식 (Windows → macOS)
- 교육 자료·Xiaozhi 가이드는 **Windows 기준**이었는데 내 환경은 macOS였다.
- 경로·툴체인·포트 이름(`/dev/cu.usbmodem*`) 차이를 맞춰 **맥에서 ESP-IDF v5.5.4로 빌드 성공**.

### 4.4 재현 가능한 실행 환경
- `demo.sh`·`flash_robot.sh`·`flash_sensor.sh`로 **포트 자동 탐색 → 플래시 → 모니터**를 스크립트화.

## 5. 단계별 브링업 기록 (00 → 06)

| 단계 | 폴더 | 배운 것 / 검증한 것 | 최종 사용 |
|---|---|---|---|
| 00 | `00_board_test` | FreeRTOS `app_main` + 시리얼 출력 — 보드 생존 확인 | ✅ |
| 01 | `01_oled_test` | I2C로 OLED 초기화·출력 | 선택 |
| 02 | `02_servo_test` | I2C로 PCA9685 서보 제어 (약 312줄) | ❌ 제거 |
| 03 | `03_walk_test` | 서보 시퀀스로 4족 보행 게이트 (약 463줄) | ❌ 제거 |
| 04 | `04_speaker_test` | I2S(MAX98357A) 스피커 출력 | ✅ |
| 05 | `05_mic_test` | I2S(INMP441) 마이크 입력·레벨 측정 | ✅ |
| 06 | `06_mic_reaction_test` | 마이크 레벨 감지 → 스피커 톤 반응 (약 477줄) | ✅ |

> `02`·`03`은 키트 원본의 보행 기능 탐색이었고, **자세 코치 목적에 불필요해 최종 제거**했다.
> 실제 코치봇은 **기울기 센서(자세) + `04`~`06` 오디오 경로 + Xiaozhi AI**로 구성된다.

## 6. 트러블슈팅 & 함정

| 함정 | 증상 | 해결 |
|---|---|---|
| **읽기전용 폴더** | 빌드/수정 실패 | 쓰기 가능한 위치로 복사 후 진행 |
| **`IDF_TARGET` 환경변수** | 엉뚱한 칩 대상 빌드 | 실행 전 `unset IDF_TARGET` |
| **시리얼 포트 이름** | 보드를 못 찾음 | macOS는 `/dev/cu.usbmodem*` — 스크립트로 자동 탐색 |
| **마이크/스피커 I2S 충돌** | 동시 사용 시 오류 | `I2S_NUM_0` 하나를 마이크↔스피커 **번갈아** 사용 |
| **불필요한 보행 하드웨어** | 서보/PCA9685가 목적과 무관 | **서보 제거**, 보행 코드 데드 처리 |

## 7. 회고 & 앞으로 학습할 것

### 배운 것
- 하드웨어를 바닥부터 올려본 경험(I2S 오디오, 기울기 센서, AI 연동).
- **필요 없는 기능(보행)을 인식하고 걷어내는 설계 판단** — 더하기보다 빼기가 중요할 때가 있다.
- 문제를 잘게 쪼개 단계별로 검증하는 습관.
- AI 협업으로 빠르게 완주하되, **동작 검증·디버깅은 결국 내 몫**.

### 아직 부족한 것 (정직하게)
- I2S를 "동작하게" 했지만 **버스 타이밍·클럭·비트포맷의 원리**까지 완전히 이해한 건 아니다.
- 기울기 센서의 자세 판정 로직을 더 견고하게(노이즈·임계값) 다듬을 여지가 있다.
- FreeRTOS **태스크/우선순위**의 깊은 이해가 필요하다.

### 학습 계획
1. **C 포인터·메모리** 기본기
2. **I2S / I2C 프로토콜 원리** — 데이터시트 직독
3. **FreeRTOS** 태스크·큐·세마포어
4. **센서 신호 처리** — 임계값·필터링

> **원칙**: 바이브 코딩으로 만든 코드를 한 줄씩 다시 읽고, 이해 안 되는 개념은 따로 공부해 노트로 남긴 뒤,
> **도구 없이 축소판을 다시 구현**해 검증한다.

## 8. 실행 방법 (재현)

```bash
source ~/.espressif/tools/activate_idf_v5.5.4.sh
unset IDF_TARGET

# 데모 (펌웨어가 이미 구워진 경우 — 모니터만)
bash ~/Desktop/0629_busan_ai/demo.sh          # 종료: Ctrl + ]

# 오디오 단계 테스트 (예: 마이크 반응)
cd ~/Desktop/0629_busan_ai/06_mic_reaction_test
idf.py -p /dev/cu.usbmodem* flash monitor
```

> `firmware/`는 Xiaozhi 오픈소스 펌웨어 이식본. `build/`·`managed_components/`는 저장소에서 제외했으니
> `idf.py reconfigure`로 의존성을 다시 받은 뒤 빌드한다.
> `02_servo_test`·`03_walk_test`는 **최종 미사용(키트 보행 원본)** — 학습 기록으로만 남겨둔다.
